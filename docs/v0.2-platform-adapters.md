# v0.2 Platform Adapters

AgentPulse v0.2 adds two intentionally different integration levels.

## Claude Code

Claude Code is a precise integration based on documented command hooks. Hook
JSON arrives on stdin and is translated as follows:

| Claude event                      | AgentPulse status    |
| --------------------------------- | -------------------- |
| `SessionStart`                    | `running`            |
| `UserPromptSubmit`                | `running`            |
| `PreToolUse`                      | `using_tool`         |
| `PostToolUse`                     | `running`            |
| `PermissionRequest`               | `waiting_permission` |
| `Notification(permission_prompt)` | `waiting_permission` |
| `Notification(idle_prompt)`       | `waiting_input`      |
| `Stop`                            | `completed`          |
| `StopFailure(rate_limit)`         | `rate_limited`       |
| other `StopFailure`               | `failed`             |

`SessionEnd` is ignored. The hook itself has no access to current daemon state,
so emitting `idle` could replace a more useful completion or failure state.

The adapter reads only the fields needed for identity and a safe description.
It does not forward transcripts, tool inputs, or the complete payload.

See [Claude Code setup](setup-claude-code.md).

## Codex CLI

Codex is a narrow integration based on its documented external `notify`
program. That interface currently documents `agent-turn-complete`, which maps
to `completed`.

The adapter uses `thread-id` as the external session ID, `cwd` as project path,
and at most 200 characters of `last-assistant-message`. It ignores complete
`input-messages`.

This interface does not expose enough information to claim Codex running,
input-waiting, or permission-waiting state.

See [Codex setup](setup-codex.md).

## Delivery behavior

Both ingest commands submit the normalized event to `POST /events`. Invalid
JSON, unsupported events, and daemon connection failures produce a safe
warning and exit code zero so the platform integration is non-blocking.

## Notifier outputs

The daemon supports `console`, `os`, and `none`. Existing notification policy
continues to notify only:

- `waiting_input`
- `waiting_permission`
- `completed`
- `failed`
- `rate_limited`

OS output is optional. Native notification runtime failures are reduced to
console warnings and do not fail ingestion.
